## 1. Governing Framework

Weber does not need a new project-management methodology invented for it — one is already mandated. The **Phase Execution Framework** (`PROJECT_CONSTITUTION.md` Part 9, restated in `DEVELOPMENT_GUIDE.md`) governs every unit of work at the project level:

> Phase Planning → Architecture Review → Project Owner Approval → Implementation → Internal Validation → Quality Assurance → Documentation → Project Owner Review → Acceptance → Next Phase.

No phase may begin before the previous one is formally accepted. Everything in this document describes *what* goes into that framework and in *what order* — never a replacement for it.

The **Documentation Before Implementation Gate** (`DECISIONS.md` WD-066) applies to every one of the 54 documented components: none may be implemented before its documentation entry has been reviewed and approved. All 54 have completed that stage. Forty-three have since entered Implementation — see `docs/components/10_MASTER_INDEX.md` for the authoritative per-component breakdown; this document tracks sequencing, not implementation status.

Tier 1 — Foundational           Layout (9), Utilities (4)

This workstream should run continuously alongside Tier 1–3 implementation, not as a single upfront gate.

## 8. Recommended Implementation Strategy

**Milestone / Phase-based** — not a new choice, but the cadence the Constitution already mandates (Part 9). `PROJECT_ROADMAP.md` confirms it directly: "Milestones will be added here as each Phase Planning report is approved, per the Phase Execution Framework." Each milestone corresponds to a natural boundary implied by the dependency tiers: Tier 1 complete → Tier 2 complete → Tier 3 complete → first working route (Contact) → Home (complete, accepted) → remaining routes → launch readiness.

**Feature-based** organization is already approved as a *code-organization* principle (`ARCHITECTURE.md` §5) and applies within each milestone — complementary, not competing.

**Sprint-based** is not supported by any approved documentation: no sprint cadence, team velocity, or fixed iteration length exists anywhere, and `PROJECT_ROADMAP.md`'s own Milestones section states target dates are `[PLACEHOLDER]`.
