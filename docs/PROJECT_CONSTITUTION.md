# PART 1 — EXECUTIVE FOUNDATION

It defines the product vision, engineering standards, UX philosophy, design principles, communication rules, governance model, and long-term strategic direction of Weber.

This constitution is the highest authority governing every design, engineering, UX, architecture, and implementation decision made throughout the lifetime of this project.

If any future instruction conflicts with this document, this constitution takes precedence until the project owner explicitly approves a change.

# PART 5 — ENGINEERING CONSTITUTION

---

# Engineering Philosophy

Engineering is not the act of writing code.

Engineering is the process of designing systems that remain reliable, maintainable, scalable, and understandable for many years.

Every technical decision must prioritize long-term quality over short-term convenience.

The objective is not to finish quickly.

The objective is to build correctly.

---

# Engineering Mission

Build Weber as a production-grade software product.

The project should be maintainable by a professional engineering team without requiring major architectural rewrites.

The codebase should remain understandable years after its creation.

---

# Engineering Values

Every engineering decision must support one or more of the following:

Reliability

Maintainability

Scalability

Performance

Security

Readability

Reusability

Consistency

Developer Experience

If a decision weakens these values, it must be reconsidered.

---

# Code Quality

Every line of code must be:

Readable.

Intentional.

Minimal.

Self-explanatory.

Consistent.

Production-ready.

The code should explain itself before requiring comments.

Comments should explain "why", not "what".

---

# Architecture Principles

The architecture must be:

Modular.

Layered.

Composable.

Extensible.

Future-proof.

Avoid tightly coupled systems.

Every feature should remain independently maintainable whenever possible.

---

# Separation of Concerns

Each module should have one clear responsibility.

Examples:

UI components render.

Business logic processes.

Database layer stores data.

API layer communicates.

Validation verifies input.

Authentication manages identity.

Each responsibility belongs in its own layer.

---

# Single Responsibility Principle

Every component.

Every function.

Every module.

Every service.

Should have one reason to change.

Avoid creating files that solve multiple unrelated problems.

---

# Reusability

If functionality is expected to be reused more than once, extract it.

Avoid duplication.

Prefer reusable abstractions over copied implementations.

Do not over-engineer.

Balance simplicity with flexibility.

---

# Naming Standards

Names must describe intent.

Avoid abbreviations.

Avoid vague names.

Examples:

createProject()

isAuthenticated()

calculateEstimate()

getPortfolioProjects()

Names should explain purpose immediately.

---

# File Organization

Project structure should remain predictable.

Files should always exist where experienced developers expect them.

Avoid deep nesting.

Avoid oversized directories.

Organize by feature whenever practical.

---

# Component Philosophy

Components should be:

Reusable.

Independent.

Predictable.

Accessible.

Small.

Composable.

Avoid giant components.

Split complexity into smaller units.

---

# State Management

Keep state local whenever possible.

Promote state only when necessary.

Avoid unnecessary global state.

Global state should represent shared application data.

Not UI convenience.

---

# API Design

APIs should be:

Consistent.

Versionable.

Secure.

Predictable.

Well validated.

Every endpoint should return meaningful responses.

Never expose internal implementation details.

---

# Database Philosophy

The database represents business truth.

Never design around convenience.

Design around data integrity.

Normalize where appropriate.

Avoid unnecessary duplication.

Maintain referential integrity.

---

# Validation

Never trust user input.

Validate:

Client side.

Server side.

Database level when appropriate.

Security begins with validation.

---

# Error Handling

Errors should never surprise developers.

Every error should:

Explain the problem.

Explain the cause.

Suggest the solution.

Avoid generic messages.

Never silently fail.

---

# Logging

Logs exist for developers.

Logs should provide enough information to diagnose issues.

Never expose sensitive information.

Never log secrets.

Never log passwords.

---

# Security

Security is a requirement.

Not a feature.

Follow secure defaults.

Protect:

Authentication.

Authorization.

API routes.

Sensitive data.

Environment variables.

Uploads.

Cookies.

Sessions.

Database access.

Never expose secrets.

Never hardcode credentials.

Never bypass validation.

---

# Performance

Performance is part of engineering quality.

Optimize:

Rendering.

Images.

Fonts.

JavaScript bundles.

Database queries.

Network requests.

Animations.

Do not optimize prematurely.

Measure before optimizing.

---

# Accessibility

Accessibility is mandatory.

Support:

Keyboard users.

Screen readers.

Proper semantic HTML.

Color contrast.

Visible focus states.

ARIA only when necessary.

Accessibility should be built in.

Not added later.

---

# SEO

Every public page should include:

Meaningful titles.

Meta descriptions.

Open Graph metadata.

Structured data where appropriate.

Semantic HTML.

Proper heading hierarchy.

Clean URLs.

Fast loading.

SEO is part of product quality.

---

# Dependencies

Every dependency increases maintenance cost.

Before adding a package, verify:

Is it actively maintained?

Is it necessary?

Can native platform capabilities solve the problem?

Prefer fewer dependencies.

---

# Documentation

Important architectural decisions must be documented.

Future developers should understand:

Why the decision exists.

What alternatives were considered.

What trade-offs were accepted.

Documentation is part of engineering.

---

# Testing Philosophy

Critical functionality should be testable.

Code should be written with testing in mind.

Avoid tightly coupled implementations that prevent testing.

---

# Version Control

Commits should represent logical units of work.

Avoid unrelated changes in the same commit.

Write meaningful commit messages.

Every commit should improve the project.

---

# Code Reviews

Before considering work complete, verify:

Does the implementation solve the problem?

Can it be simplified?

Is it reusable?

Is it secure?

Is it accessible?

Does it match the constitution?

If improvements exist, implement them before requesting approval.

---

# No Assumptions Policy

Never assume missing requirements.

If anything is unclear:

Stop.

Explain.

Present available options.

Recommend the preferred solution.

Wait for approval.

Engineering decisions affecting product direction require explicit authorization.

---

# Communication Workflow

For every implementation task:

1. Explain the objective.

2. Explain why the work is necessary.

3. Explain which files will change.

4. Explain possible risks.

5. Wait for approval if required.

6. Implement.

7. Summarize the implementation.

8. Explain architectural impact.

9. Suggest future improvements.

Never modify the project silently.

---

# Technical Debt

Avoid temporary solutions unless explicitly approved.

If a temporary solution becomes necessary:

Clearly label it.

Explain the reason.

Document how it should be replaced.

---

# Scalability

Every new feature should be evaluated against future expansion.

Ask:

Will this architecture still work when Weber becomes:

An educational platform?

A SaaS platform?

A customer portal?

A learning ecosystem?

If the answer is no,

reconsider the implementation.

---

# Engineering Decision Framework

Before implementing any solution, ask:

Is it maintainable?

Is it scalable?

Is it secure?

Is it readable?

Is it reusable?

Is it testable?

Does it improve the project?

Does it comply with this constitution?

Only proceed when the answer is yes.

---

# Final Engineering Principle

Write code as if another senior engineer will maintain it for the next ten years.

The objective is not to impress other developers.

The objective is to build software that remains understandable, reliable, and valuable long after it is written.

# Design Philosophy

The Weber Design System exists to create a unified, scalable, and maintainable user interface.

Every interface element must feel like it belongs to the same product.

Consistency builds trust.

Inconsistency destroys it.

The design system is not a collection of components.

It is a language.

Every page speaks this language.

---

# Design Goals

The Design System must achieve the following:

Visual Consistency

Scalability

Accessibility

Maintainability

Developer Efficiency

Fast Decision Making

Premium Appearance

Long-Term Stability

---

# Design Principles

Every UI element should be:

Simple

Intentional

Consistent

Predictable

Accessible

Elegant

Minimal

Avoid decoration.

Design should communicate purpose.

---

# Design Tokens

The Design System must be driven by tokens rather than hardcoded values.

Tokens should define:

Colors

Spacing

Typography

Border Radius

Elevation

Animation

Transitions

Breakpoints

Opacity

Sizing

Future changes should require modifying tokens—not components.

---

# Color System

Official Colors

Background

#0B0C0F

Primary

#FFFFFF

Secondary

#D4D4D8

Accent

#71717A

Additional colors may be introduced only when they solve a real usability problem.

Never introduce decorative colors.

Colors communicate function.

Not decoration.

---

# Color Usage

Background should dominate.

Primary should attract attention.

Secondary should support content.

Accent should be used sparingly.

The interface should never become colorful.

White space is more valuable than additional colors.

---

# Typography System

Typography must communicate clarity.

Hierarchy should be obvious without effort.

Use a modern sans-serif typeface.

The font family should remain consistent across the entire product.

Typography should scale predictably.

Avoid arbitrary font sizes.

Maintain generous line-height.

Optimize readability before aesthetics.

---

# Typography Hierarchy

Each page should have one H1.

Subheadings should follow logical structure.

Body text should remain comfortable for long reading sessions.

Captions should support—not replace—content.

Never skip heading levels.

---

# Grid System

Use a consistent layout grid.

Layouts should align predictably.

Content should never appear randomly positioned.

Grid consistency creates visual rhythm.

---

# Spacing System

Spacing should follow a mathematical scale.

Avoid random margins.

Avoid arbitrary padding.

Consistent spacing creates visual harmony.

Whitespace is intentional.

Whitespace improves comprehension.

Never fill empty space simply because it exists.

---

# Layout Principles

Every page should feel balanced.

Content width should remain comfortable for reading.

Large screens should not stretch content unnecessarily.

Layouts should breathe.

The eye should naturally flow from one section to another.

---

# Component Philosophy

Components should solve one problem.

Buttons should trigger actions.

Cards should group information.

Dialogs should request attention.

Badges should communicate status.

Inputs should collect information.

Avoid multi-purpose components whenever possible.

---

# Component Consistency

Buttons with identical purposes must always look identical.

Cards should follow consistent spacing.

Forms should behave consistently.

Navigation patterns should remain unchanged across pages.

Users should never relearn the interface.

---

# Icons

Icons should support understanding.

Icons should never replace clear labels.

Use one icon family throughout the project.

Avoid decorative icon usage.

---

# Border Radius

Corner radius should remain consistent.

Different radius values should communicate different intentions.

Avoid excessive rounding.

The interface should feel modern rather than playful.

---

# Elevation

Shadows should communicate hierarchy.

Avoid dramatic shadows.

Depth should be subtle.

Use elevation only when necessary.

Flat design is preferred unless depth improves usability.

---

# Motion System

Motion should communicate:

State changes.

Continuity.

Feedback.

Hierarchy.

Motion should never exist purely for decoration.

Every animation must answer:

"What information does this communicate?"

---

# Animation Principles

Animations should be:

Smooth.

Short.

Natural.

Predictable.

Respect user preferences for reduced motion.

Never delay interaction unnecessarily.

---

# Responsive Design

The Design System must work across:

Mobile

Tablet

Laptop

Desktop

Large Displays

Design mobile-first.

Enhance progressively.

Never sacrifice usability for aesthetics.

---

# Forms

Every form should behave consistently.

Labels remain visible.

Validation is immediate.

Errors are understandable.

Required fields are obvious.

Success states are reassuring.

---

# Interactive States

Every interactive element should clearly communicate:

Default

Hover

Focus

Pressed

Disabled

Loading

Success

Error

Users should always understand the current state.

---

# Empty States

Every empty state should:

Explain why it is empty.

Guide users toward the next action.

Maintain the brand personality.

Empty states are opportunities to educate.

---

# Loading States

Never leave users wondering.

Show meaningful loading indicators.

Maintain layout stability.

Avoid sudden content shifts.

Perceived performance matters.

---

# Error States

Errors should:

Explain the problem.

Explain why it happened.

Suggest the next step.

Remain calm.

Never blame the user.

---

# Accessibility

The Design System must include:

Proper contrast.

Keyboard accessibility.

Visible focus indicators.

Readable font sizes.

Accessible spacing.

Touch-friendly targets.

Accessibility is built into the system—not added later.

---

# Dark Theme Philosophy

Weber's identity is built around its dark theme.

Dark mode is the primary experience.

If a light theme is introduced in the future,

it must respect the same design language.

Dark mode should never feel like an afterthought.

---

# Future Scalability

Every future component should inherit this Design System.

Never create isolated visual styles.

Never redesign individual pages independently.

The Design System evolves.

The identity remains consistent.

---

# Design Review Checklist

✓ Is accessibility respected?

✓ Is visual hierarchy clear?

✓ Is unnecessary complexity removed?

# Final Design Principle

People should never notice the design system.

They should notice how effortless the experience feels.

Consistency is invisible.

That invisibility is the highest level of design quality.

Whether "React components" should be read literally (React is the confirmed frontend framework) or generally (the same separation principle applies to whatever component framework is eventually chosen) requires Project Owner clarification — see the open question raised alongside this amendment.

Principle regardless of framework: business, marketing, and product content (copy, offerings, pricing language, testimonials, etc.) must be kept separate from UI component code unless the content is genuinely tied to the UI itself (e.g., a button label that only exists because of that specific interaction). This keeps content editable, reusable, and independent of implementation.

### 2. Single Source of Truth Rule

> "Data should always have a single source of truth."

Extends: Part 5 — Database Philosophy, State Management.

No piece of data should be duplicated across multiple stores, components, or state containers in a way that allows copies to drift out of sync. Every piece of data has exactly one authoritative origin; everything else derives from or references it.

---

### 3. Security Decision Authority Rule

> "Security is a requirement, not a feature."
>
> "AI may generate code, but it may never decide security."

Extends: Part 1 — Decision Authority, Part 5 — Security.

"Security is a requirement, not a feature" reaffirms the principle already stated in Part 5 — Security ("Security is a requirement. Not a feature."). It is restated here to anchor the new rule that follows.

New rule: the AI may write and implement security-related code (authentication flows, validation, access control, etc.), but it may never independently choose or decide the underlying security approach — authentication method, encryption strategy, authorization model, session strategy, or any other security-critical architecture. Those choices must be explicitly proposed with trade-offs and approved by the Project Owner before implementation, consistent with the Decision Hierarchy in Part 7 (Engineering Decision → AI Recommendation → Project Owner Approval).

### 4. Performance Trade-off Rule

> "Performance is part of the user experience."
>
> "No feature may be added if it noticeably degrades performance without delivering equivalent user value."

Extends: Part 4 — Performance Equals UX, Part 5 — Performance, Part 8 — Performance Review.

"Performance is part of the user experience" reaffirms the principle already stated in Part 4 — Performance Equals UX ("Performance is part of the user experience."). It is restated here to anchor the new rule that follows.

New rule: performance cost is now an explicit gate on feature approval. Before any feature is added, its performance impact must be weighed against the user value it delivers. If a feature noticeably degrades performance without delivering equivalent value to the user, it must not be implemented — this applies during Phase Planning and the Performance Review step of the Architect Review Mindset (Part 8), not only after the fact.
